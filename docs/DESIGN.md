# Design

This is the working design doc. It captures the shape of the system as we currently understand it; anything marked **[TBD]** is an open question and should also live in [`OPEN_QUESTIONS.md`](./OPEN_QUESTIONS.md).

---

## Core idea: close the loop with a filesystem graph

The existing `/dev` and `/dev-plan` commands are excellent in isolation — `/dev-plan` does research → PRD → party-mode refinement → emits epic slugs; `/dev` takes a slug and implements it with full discipline (CI, coverage, adversarial review). But **the user is the glue** between them. You run `/dev-plan`, you read the output, you decide which epic to run, you run `/dev <slug>`, you wait, you pick the next one.

devx replaces the human glue with a **filesystem graph**:

- Every unit of work is a file (`dev/dev-<hash>-<timestamp>-<slug>.md`).
- Every backlog is a file (`DEV.md`, `TEST.md`, …) that references spec files.
- Every agent reads the backlog for its role, picks the top runnable item, and appends its status to the spec file as it works.
- A supervisor agent (`/devx-manage`) rebalances the backlog, resolves conflicts, decides what runs next, and keeps workers alive across context rot.

Nothing persists outside the repo. `git log` is the audit trail. `grep` is the query engine.

## The eight backlog files

All live at the repo root. Each is a markdown file whose body is a bulleted list of spec-file references, plus some human-readable context at the top.

### `DEV.md` — features to build

Written by: PlanAgents (new items), DevAgents (progress updates), ManageAgent (reprioritization).
Read by: DevAgents (pick next), TestAgents (know what just shipped).

Each entry:
```markdown
- [ ] `dev/dev-a3f2b9-2026-04-23T14:22-add-oauth-google.md`
  - Title: Add Google OAuth to login
  - From: plan/plan-9c1d4a-epic-auth.md
  - Status: ready
  - Blocked by: —
  - Owner: —   (picked up → DevAgent-7 when claimed)
```

Statuses: `ready`, `in-progress`, `blocked`, `in-review`, `done`, `deleted`.

#### Checkbox conventions

The checkbox at the start of each backlog entry is a glanceable mirror of the `Status:` field. ManageAgent keeps them in sync on every reconcile tick — hand-edit either side and the other follows.

| Marker | Status | Worker behavior |
|---|---|---|
| `[ ]` | `ready` — claimable | `/devx` no-args picks the top one |
| `[/]` | `in-progress` — claimed by a worker | skip; spec lock is held |
| `[-]` | `blocked` — waiting on INTERVIEW / MANUAL / dependency | skip; Manager polls for unblock |
| `[x]` | `done` — merged to `develop` | skip; archived after `storage.spec_archive_after_days` |
| line wrapped in `~~…~~` | `deleted` / abandoned | skip; kept for audit |

GitHub-flavored Markdown only renders `[ ]` and `[x]` natively; the others render as bracket text. Deliberate — the plain-text fallback is still informative when scanning, and the Status field carries the authoritative state.

#### `/devx` with no args = auto-pick

Calling `/devx` without a slug picks the top runnable item in `DEV.md`:

1. Scan `DEV.md` top-down for entries marked `[ ]`.
2. Skip any whose spec frontmatter's `Status:` ≠ `ready` (defensive — backlog might be stale relative to the spec).
3. Skip any with unsatisfied `Blocked-by:` references.
4. Acquire `spec-<hash>.lock` (atomic O_EXCL); on contention, move to the next candidate.
5. Flip the checkbox `[ ]` → `[/]`, write `status: in-progress` and an `<ts> — claimed by <agent-id>` line into the spec status log, and proceed.

Pass `/devx <hash>` or `/devx <slug>` to override and pick a specific item. The auto-pick is the default because it's how Manager spawns workers — the no-args invocation is the same code path a human invokes.

### `PLAN.md` — planning work in flight

Written by: PlanAgents, ManageAgent.
Read by: PlanAgents (avoid re-planning the same thing), user.

Tracks active planning: which epics are being drafted, what research is outstanding, which party-mode workshops are scheduled.

### `TEST.md` — test work

Written by: DevAgents (declares coverage gaps on commit), TestAgents (progress), FocusAgent (user flow gaps).
Read by: TestAgents.

```markdown
- [ ] `test/test-f8e2a1-oauth-callback-coverage.md`
  - Target: services/auth/oauth.py — google_callback handler
  - Gap: no test for expired-state rejection
  - From: dev-a3f2b9 (auto-generated by DevAgent on merge)
```

### `DEBUG.md` — things that are broken

Written by: anyone (DevAgent on CI red, FocusAgent on user friction, user manually, TestAgent on flake).
Read by: DebugAgents.

```markdown
- [ ] `debug/debug-c4a7e2-2026-04-23T09:14-login-500-on-safari.md`
  - Source: production logs — 47 occurrences in last 24h
  - Symptom: 500 on POST /auth/login from Safari 17
  - First seen: 2026-04-22
  - Repro: [TBD by DebugAgent]
```

### `FOCUS.md` — user signals

Written by: FocusAgent (synthesis of telemetry + interviews), user (direct feedback).
Read by: PlanAgents (what to plan next), DebugAgents (known user pain).

Not individual items — instead, rolling summaries:
- Top 5 user friction points this week
- Features requested ≥3 times
- Flows with >20% abandonment
- Paths where session time is anomalously high

Each summary points at the raw telemetry query that produced it (so it can be re-run).

### `INTERVIEW.md` — questions for the user

Written by: any agent (when blocked on a decision a human must make).
Read by: the user.

```markdown
- [ ] Q#4 (from DevAgent-7 on dev-a3f2b9)
  - Context: implementing Google OAuth callback
  - Question: Should the redirect URI use our root domain (example.com/auth/callback) or a dedicated subdomain (auth.example.com/callback)?
  - Blocks: dev-a3f2b9, plan-9c1d4a (epic-auth)
  - Options: (a) root domain, (b) subdomain, (c) make configurable
  - Agent recommendation: (c)
```

The user answers inline by checking the box and adding a `→ Answer: ...` line. ManageAgent detects answered questions and unblocks the waiting items on its next reconcile tick.

### `MANUAL.md` — things only the user can do

Written by: any agent (when action requires a human — approve a cloud resource, add a secret, sign in to a third-party service, review a sensitive PR).
Read by: the user.

```markdown
- [ ] M#2 (from DevAgent-7 on dev-a3f2b9)
  - Action: Add GOOGLE_OAUTH_CLIENT_SECRET to GitHub Actions secrets
  - Why: CI will fail integration tests without it
  - Blocks: dev-a3f2b9 merging
  - How: https://github.com/<org>/<repo>/settings/secrets/actions
```

Distinction from `INTERVIEW.md`: interview = decision, manual = action. Agents can recommend either, but users execute `MANUAL.md` items.

### `LESSONS.md` — self-healing proposals

Written by: LearnAgent (detected patterns), user (hand-crafted lessons via `/devx-learn --add`).
Read by: ManageAgent (when deciding idle-capacity scheduling), user (approve/reject pending lessons), mobile app (inbox).

Each entry points at a `learn/learn-<hash>-<ts>-<slug>.md` spec file that carries the full evidence chain: which triggers produced the lesson, what target it proposes to edit, the proposed diff, the confidence level, and (once applied) the resulting commit.

See [`SELF_HEALING.md`](./SELF_HEALING.md) for the complete contract — trigger signals, confidence gates per write target, canary runs for skill/prompt changes.

## Spec file convention

Every item in every backlog points at a file. The path encodes type + identity + time + human-readable name:

```
<type>/<type>-<hash>-<timestamp>-<slug>.md

Examples:
  dev/dev-a3f2b9-2026-04-23T14:22-add-oauth-google.md
  test/test-f8e2a1-2026-04-23T15:50-oauth-callback-coverage.md
  debug/debug-c4a7e2-2026-04-23T09:14-login-500-on-safari.md
  plan/plan-9c1d4a-2026-04-20T10:00-epic-auth.md
  focus/focus-e1b5d8-2026-04-22T00:00-weekly-friction-summary.md
```

- `<type>` — one of `dev`, `plan`, `test`, `debug`, `focus`, `learn`, `qa`. Implies the agent class.
- `<hash>` — 6 random hex chars. Unique handle; short enough to reference inline.
- `<timestamp>` — ISO 8601 local time at creation, minute precision. Lets you sort chronologically without a DB.
- `<slug>` — kebab-case human description, trimmed to ~50 chars.

### Branch field — derived from `devx.config.yaml`

The spec's `branch:` frontmatter is a **derivation**, not a fixed default. The planner computes it from `devx.config.yaml`:

- `git.integration_branch: null` (single-branch model) → `branch: <git.branch_prefix>dev-<hash>` (e.g. `feat/dev-aud101`).
- `git.integration_branch: <name>` (e.g. `develop`) → `branch: <name>/<git.branch_prefix>dev-<hash>` (or whatever the project's documented convention is).

Hardcoding `develop/dev-<hash>` (or any other fixed prefix) into spec frontmatter caused every Phase 0 story to "correct branch" on claim — see `LEARN.md § epic-bmad-audit / epic-config-schema / epic-cli-skeleton / epic-os-supervisor-scaffold`.

### Source-of-truth precedence (when artifacts disagree)

Real conflicts have already shown up between a story's spec ACs and its parent epic's "locked decisions" (cfg202: spec said XDG-Linux, epic said `~/.devx/`) and between spec ACs and party-mode minutes (cli302: party-mode wanted a `preview:` line, spec said stderr "match exactly"). The precedence rule:

1. **Story spec acceptance criteria** — highest. The spec is what `/devx` claims and ships.
2. **Epic file's "Locked decisions" section** — overrides defaults but loses to the spec.
3. **Plan-spec frontmatter** (`mode`, `project_shape`, `thoroughness`).
4. **`devx.config.yaml`** defaults.
5. **Skill-prompt or template defaults** — lowest.

When `/devx` finds a conflict, follow the spec, and in the same PR file a doc/spec correction so the lower-precedence source is updated to match (or marked as the deliberate exception with a one-line rationale).

```markdown

from: plan/plan-9c1d4a-epic-auth.md
spawned:
  - test/test-f8e2a1-oauth-callback-coverage.md

owner: DevAgent-7
branch: feat/dev-a3f2b9        # derived from devx.config.yaml; see § Branch field above
worktree: .worktrees/dev-a3f2b9
---

## Goal
<what this item does>

## Acceptance criteria
- [ ] user can click "Sign in with Google"
- [ ] callback validates state parameter
- [ ] valid callback creates/fetches user and returns JWT
- [ ] expired state returns 400

## Technical notes
<anything the agent figured out during implementation>

## Status log
- 2026-04-23T14:22 — created by PlanAgent-3
- 2026-04-23T14:25 — claimed by DevAgent-7
- 2026-04-23T14:28 — blocked on INTERVIEW q#4 (redirect URI)
- 2026-04-23T15:10 — unblocked (user answered: c, make configurable)
- 2026-04-23T15:45 — implementation complete, PR #142 opened
- 2026-04-23T15:50 — test gaps written to test/test-f8e2a1-...

## Links
- PR: https://github.com/.../pull/142
- CI: https://github.com/.../actions/runs/...
```

The status log is **append-only**. Agents add lines; they don't rewrite them. This is the request history — where a thing came from, where it went.

## Agent roles

Every agent class maps 1:1 to a slash command, but the slash commands can also be invoked manually. ManageAgent spawns agents behind the scenes.

### `PlanAgent` — `/devx-plan`

- Input: a raw idea, or an entry in `PLAN.md`, or a signal from `FOCUS.md`.
- Work: research → PRD update → architecture update → epic chunking → party-mode refinement.
- Output: new spec files under `dev/`, new entries in `DEV.md`, questions in `INTERVIEW.md`, cross-refs from `FOCUS.md`.
- Inherits the discipline of the current `/dev-plan` command — the seven-phase loop, party-mode for every epic, three-layer coverage, YOLO vs. ask rules.

### `DevAgent` — `/devx`

- Input: next `[ ]` item in `DEV.md` (auto-picked when called without a slug; see [§Checkbox conventions](#checkbox-conventions)). `/devx <hash>` or `/devx <slug>` overrides the auto-pick.
- Work: claim (`[ ]` → `[/]`, status `in-progress`, lock `spec-<hash>.lock`) → create worktree + branch → implement in red-green-refactor → self-review (adversarial) → push → wait for CI → merge (or revise) → mark `[x]` and `status: done`.
- Output: merged code, updated spec file status log, test gaps appended to `TEST.md`, debug items appended to `DEBUG.md` on CI failure.
- Inherits the discipline of the current `/dev` command — the six-phase loop, story-per-commit, check-models drift detection, pubspec bumps, CI-as-truth.

### `TestAgent` — `/devx-test`

- Input: next item in `TEST.md`.
- Work: author unit/integration/**scripted Playwright** tests, enforce 100% coverage on touched surface.
- Output: merged test code, regression items appended to `DEBUG.md` when a test reveals a bug.
- Leans on the BMAD `tea` module (test architecture) for strategy, ATDD, trace, NFR.
- **Does not** run exploratory browser QA — that's `/devx-focus`'s job via a subprocess LLM-driven browser agent. See [`QA.md`](./QA.md) for the Layer 1 / Layer 2 split.

### `DebugAgent` — `/devx-debug`

- Input: next item in `DEBUG.md`.
- Work: pull logs/metrics/DB state via observability hooks → reproduce → fix → write regression test → push.
- Output: merged fix, test added to `TEST.md` progress, status log updated on the debug spec.

### `FocusAgent` — `/devx-focus` (and `/devx-focus-group`)

- **Simulated mode:** consults the persistent persona panel in `focus-group/personas/` about a proposed change. Invoked during `/devx-plan` after party-mode, before `develop → main` promotion, and on demand.
- **Empirical mode:** ingests real user telemetry + `INTERVIEW.md` answers + explicit feedback + `qa/*.md` results from exploratory browser runs; synthesizes patterns, cross-references with panel predictions, and evolves persona reaction libraries.
- **Output:** `focus-group/sessions/session-*.md` (per consultation), rolling summary in `FOCUS.md`, new items in `DEV.md`/`DEBUG.md`/`INTERVIEW.md`, persona edits queued in `LESSONS.md`.
- Built on BMAD's `advanced-elicitation` "User Persona Focus Group" method (method #4) as the interaction primitive. Browser QA subprocess-spawned per [`QA.md`](./QA.md); persona prompts for QA seeded directly from `focus-group/personas/*.md`.
- Full contract: [`FOCUS_GROUP.md`](./FOCUS_GROUP.md).

### `LearnAgent` — `/devx-learn`

- Input: request graph (every spec file's status log + `git log` + recorded CI outcomes + skill-edit log).
- Work: scan for trigger signals (repeated blockers, repeated CI failures, user edits after agent writes, recurring INTERVIEW answers, repeated PR-review findings), cluster into lessons, draft diffs against the chosen write target, gate by confidence + blast radius.
- Output: `LESSONS.md` entries + `learn/*.md` spec files with full evidence. Auto-applies high-confidence + low-blast lessons; queues everything else for user approval. Canary-runs agent-prompt changes before merge.
- Scheduled by ManageAgent when capacity is idle and signals have accumulated; can also be invoked directly by the user or via `/loop 24h /devx-learn`.
- Operates under the same worktree + `develop/learn-<hash>` branching rules as every other agent.

Full contract: [`SELF_HEALING.md`](./SELF_HEALING.md).

## Control plane

The agent classes above are *workers* — they consume one backlog and produce into others. Two additional agents form the **control plane** that decides what runs, supervises lifecycle, and talks to the user. Borrowing Kubernetes terms because the shape is the same:

| Agent | Kubernetes analog | Job |
|---|---|---|
| `ManageAgent` | scheduler + kubelet | Decides what to run, spawns workers, restarts on context rot, kills on timeout |
| `ConciergeAgent` | apiserver + notification gateway | Single conversational front door for the user; routes inbound; emits outbound notifications |

The filesystem graph (eight backlog files + `.devx-cache/`) is the etcd equivalent — durable, inspectable, single source of truth.

### `ManageAgent` — `/devx-manage`

The supervisor + scheduler. **Always running.** Owns one machine's worker pool. Two responsibilities, both in the same loop:

**Scheduling — what should be running:**

1. Reconcile duplicate / superseded backlog entries.
2. Detect blocks and unblocks (INTERVIEW answers, MANUAL completions).
3. Reprioritize using FOCUS signals.
4. Compute the desired roster: a list of `(spec, worker-class)` pairs that should be running, up to `capacity.max_concurrent` (see [`CONFIG.md`](./CONFIG.md)).
5. Schedule LearnAgent on idle capacity when retro signals have accumulated.

**Supervision — keeping it running:**

- **Spawns** worker subprocesses (`claude` invocations of `/devx-plan`, `/devx`, etc.) to fill the desired roster.
- **Watches** each worker's `.devx-cache/events/<agent-id>.jsonl` event stream for: explicit "stopping for context" lines, token usage hitting `manager.restart_on_token_pct`, wall-clock exceeding `manager.max_worker_age`, exit code != 0, missing heartbeat.
- **Restarts** a rotted worker by killing the subprocess and spawning a fresh `claude /devx-<role> <slug>` against the same spec file. The spec's status log carries the resume context — no continuation snippet, no human in the loop.
- **Tears down** workers whose specs reached `done` / `killed` / `blocked`; releases their worktree slot.
- **Spawns browser-QA subprocesses** on TestAgent / FocusAgent demand (Playwright, persona-driven) per [`QA.md`](./QA.md).
- **Also supervises ConciergeAgent.** Concierge is just another long-lived child; if it rots, ManageAgent restarts it the same way.

ManageAgent persists its desired roster to `.devx-cache/state/schedule.json` between ticks so a crash + restart is recoverable from disk (see [§Crash recovery](#crash-recovery)). It is intentionally a *thin* loop — small prompt, narrow tool surface (filesystem + subprocess + heartbeat) — so its own context doesn't rot fast. A dedicated GitHub Actions cron pings the heartbeat hourly and notifies the mobile app if it goes stale (see [§Staying alive](#staying-alive)).

### `ConciergeAgent` — `/devx-concierge`

The single conversational surface for the user. **Always running, supervised by ManageAgent.** Separate from ManageAgent because user-I/O cadence (interactive, blocking on the user) is incompatible with scheduling/supervision cadence (continuous, time-sliced); folding them together meant Concierge would freeze the supervisor every time the user typed.

- **Inbound**: receives user messages from CLI (`devx ask`), mobile app, or scheduled prompts. Parses intent, routes to backlog: a feature request becomes a `DEV.md` entry; a bug report becomes a `DEBUG.md` entry; a question to the agents becomes an `INTERVIEW.md` reply. Concierge does not implement — it dispatches.
- **Outbound**: subscribes to ManageAgent's high-level event stream and emits notifications via configured channels (FCM → mobile, email, webhook) per `notifications.events`. E.g.: "tell me about MANUAL items and context-rot restarts; don't tell me about phase changes."
- **Session digest**: on demand or on schedule, produces a compact "what's happening right now" — *"3 agents running. 2 manual tasks waiting. 1 PR ready for promotion."*

Concierge is intentionally minimal-context — a router and notifier, not a reasoner. Anything requiring real reasoning is delegated to a freshly spawned worker via ManageAgent. This is how Concierge avoids context rot despite being long-lived.

## Agent flow graph

The eight backlog files don't sit in isolation — they feed each other. Every agent class both *consumes* one or more backlogs and *produces* into others. This graph is the load-bearing picture of how work moves through the system:

```
                            ┌─────────────────┐
                            │  user / phone   │  raw ideas, requirements
                            └────────┬────────┘
                                     │
                                     ▼
        ┌────────────────┐    ┌─────────────┐    ┌──────────────────┐
        │  FOCUS.md      │───▶│  PLAN.md    │◀───│  INTERVIEW.md    │
        │  (telemetry,   │    │             │    │  (answered Qs    │
        │  user signal)  │    │  PlanAgent  │    │   feed planning) │
        └────────────────┘    └──────┬──────┘    └──────────────────┘
                ▲                    │ writes
                │                    ▼
                │             ┌─────────────┐    ┌──────────────────┐
                │             │  DEV.md     │───▶│  focus-group/    │
                │             │             │    │  (panel reacts)  │
                │             │  DevAgent   │    └──────────────────┘
                │             └──────┬──────┘
                │ FocusAgent          │ writes & spawns
                │ synthesizes         ▼
                │             ┌─────────────┐    ┌──────────────────┐
                │             │  TEST.md    │    │  DEBUG.md        │
                │             │             │    │                  │
                │             │ TestAgent   │    │  DebugAgent      │
                │             └──────┬──────┘    └────────┬─────────┘
                │                    │ on flake / failure │ on user repro
                │                    └────────────────────┤
                │                                         ▼
                │             ┌──────────────────────────────────┐
                │             │ retros/ — RetroAgent (per spec)  │
                │             └──────────────┬───────────────────┘
                │                            │ ≥3 concordant
                │                            ▼
                │             ┌──────────────────────────────────┐
                │             │ LESSONS.md — LearnAgent          │
                │             └──────────────┬───────────────────┘
                │                            │ writes (mode-gated)
                │                            ▼
                │             memory · CLAUDE.md · skills · config · templates
                │                            │
                └────────────────────────────┘ (next run is tighter)
```

### Reading the graph

- **Vertical = lifecycle**: ideas at top, learning at bottom; each layer's output is the next layer's input.
- **MANUAL.md and INTERVIEW.md are the user's two inboxes**: any agent can write into them; only the user can drain them.
- **The control plane reads everything**: ManageAgent (scheduler + supervisor) and ConciergeAgent (user I/O) don't appear as nodes because they sit *over* the graph rather than in it. ManageAgent picks the next runnable item per agent class given mode, shape, locks, and capacity, then spawns/respawns workers against that decision; Concierge handles user-facing routing and notifications.
- **The user can inject anywhere**: drop directly into DEV.md (idea), INTERVIEW.md (answer), DEBUG.md (bug report), MANUAL.md (acknowledged action). Agents never overwrite user-typed lines marked `[locked]`.

This graph is also rendered in `focus-group/index.md` and the mobile app's Activity tab so the user can scan "where is everything" at a glance.

## Thoroughness levels

Mode covers risk-to-user-data. Project shape covers state-of-codebase. **Thoroughness covers process depth + token budget.** Three independent axes; thoroughness is the one that controls *how much research and ceremony* every agent applies.

| Thoroughness | Per-feature token bias | What changes |
|---|---|---|
| `send-it` | minimum | PlanAgent skips party-mode unless the epic crosses ≥2 surfaces; focus-group skipped except in PROD; QA Layer 2 on-demand only; coverage advisory; RetroAgent runs but LearnAgent threshold raised to ≥5 concordant retros so the system mutates slowly when you're moving fast |
| `balanced` (default) | ~1× | Party-mode for every epic; focus-group consulted at plan + pre-promotion; QA Layer 2 nightly per mode; coverage and retros at mode defaults |
| `thorough` | 2–3× | Party-mode + advanced-elicitation pre-write; focus-group at plan + post-implementation + pre-promotion; QA Layer 2 nightly + per-PR; deeper research passes; LearnAgent threshold lowered to ≥2 concordant retros (catch more, gate harder) |

### How thoroughness combines with mode

Thoroughness is the *process* knob; mode is the *gate* knob. They stack:

| Combination | Behavior |
|---|---|
| `YOLO + send-it` | Maximum velocity. Build, test (informational), ship. Self-healing barely fires. |
| `BETA + balanced` | Default for typical solo project. |
| `PROD + thorough` | Maximum care. Every gate, every panel, every retro counted. |
| `PROD + send-it` | Allowed but flagged at init: "you're shipping to real users with minimum process — confirm." |
| `YOLO + thorough` | Allowed; rare but valid (e.g., greenfield architectural prototype). |

### Where it lives

```yaml
# devx.config.yaml
thoroughness: balanced   # send-it | balanced | thorough
```

Override per-command (`/devx --thoroughness send-it`) for one-off escapes. Override per-spec via spec frontmatter `thoroughness: thorough` for a specific epic that warrants extra care regardless of project default.

### Why a third axis

Considered collapsing thoroughness into mode (e.g., YOLO = send-it, PROD = thorough) and rejected: a YOLO prototype can still warrant thorough architectural exploration for a greenfield decision; a PROD app can warrant send-it on a trivial copy fix. Two-axis collapse forced the wrong default for at least one real workflow Leonid runs. Three is the minimum that captures it.

## Branching model

devx **recommends but does not require** a `develop/main` split. The split is the
default for non-YOLO projects and the recommended option `/devx-init` offers,
but a single-branch flow is fully supported via `git.integration_branch: null`
+ `git.protect_main: false` for users who don't want the ceremony (typical for
solo YOLO projects, prototypes, or one-off scripts).

When the split is enabled (the default for BETA / PROD):

- **`main`** = production. What's deployed. Protected (branch protection enforced, no direct pushes, no force-pushes ever). Only accepts PRs from `develop`, and only via explicit promotion.
- **`develop`** = integration. Where in-progress agent work accumulates. This is where the mobile app adds items, where agent PRs merge, where coverage gates run, where e2e tests live.
- **Feature branches** = `develop/<type>-<hash>`, one per spec file. Branched off `develop`, PR'd back into `develop`.

When single-branch is chosen (`integration_branch: null`):

- Feature branches are `<branch_prefix><type>-<hash>` off `main` (default
  prefix `feat/` instead of `develop/`).
- Agents PR back to `main` (`pr_strategy: pr-to-main`) or, in YOLO with
  explicit consent, push direct (`pr_strategy: direct-to-main`).
- The promotion gate collapses into the merge gate — there is no separate
  promotion step.
- The mobile app and user hand-edits commit directly to `main`.

The rest of this section describes the split flow because it's the recommended
shape; behavior under single-branch is the obvious simplification (skip every
"promote" step, treat every merge as a deploy).

### Branch naming

```
main                                           ← production, deployed
develop                                        ← integration, WIP
develop/dev-a3f2b9                             ← DevAgent-7 working dev-a3f2b9
develop/debug-c4a7e2                           ← DebugAgent-2 working debug-c4a7e2
develop/test-f8e2a1                            ← TestAgent-4 working test-f8e2a1
develop/plan-9c1d4a                            ← PlanAgent-3 working plan-9c1d4a
```

### Lifecycle of a unit of work

```
┌──────────────────┐    ┌───────────┐    ┌───────────────────┐    ┌────────┐
│ feature branch   │ →  │  develop  │ →  │  promotion gate   │ →  │  main  │
│ develop/dev-XXX  │    │ (CI green)│    │  (extended gates) │    │(deploy)│
└──────────────────┘    └───────────┘    └───────────────────┘    └────────┘
      agent works           auto-merge        user-approved
      in worktree           after CI          or Manager-decided
                            + review
```

Five stages:

1. **Agent works in its feature branch's worktree.**
2. **Agent opens PR → `develop`.** CI runs; code review runs; coverage gate runs.
3. **Auto-merge to `develop`** once gates pass. ManageAgent can merge; no human required for this step by default.
4. **Promotion gate.** Separate, slower cadence. Runs extended checks: integration tests against staging, browser-agent QA re-walks critical user flows, 24h soak on develop, FOCUS telemetry shows no regressions. Can be user-gated or Manager-gated (configurable).
5. **Promote `develop → main`** via a squash-merge or merge commit. This push triggers the deploy pipeline.

### What each branch means for agents

- **DevAgents, DebugAgents, TestAgents, PlanAgents** → always branch off `develop`, always PR into `develop`. They literally cannot target `main`.
- **The mobile app** → commits directly to `develop` (backlog appends) or opens branches off `develop`. Never touches `main`.
- **The user hand-editing backlog files** → edits `develop` (that's the default branch agents and the phone read from). When the user wants to promote, they run `/devx-manage --promote` or merge the PR manually.
- **A dedicated PromotionAgent** (new, resolves part of OPEN_QUESTIONS #7) → runs the promotion gate logic. Opens the `develop → main` PR, runs extended checks, merges when they pass. User can veto by closing the PR.

### Why this shape

- **The phone can't brick production.** Mobile writes hit `develop`; if they're bad, agent CI catches it before `main`.
- **Agents iterate fast on `develop`.** No wait for production deploy on every DevAgent merge.
- **Deploy cadence is a separate decision from merge cadence.** You can merge 30 agent PRs to `develop` before deciding any of them are ready to ship.
- **Rollback is `git revert` on `main`, not on the whole work graph.** Production is isolated from WIP.
- **CI configuration is layered:** `develop` CI = lint + unit + integration + coverage. `main` CI = deploy + smoke. Different jobs, different budgets.

### What `/devx-init` sets up for this

- Creates `develop` branch if absent, sets it as the default branch for the repo.
- Enables branch protection on `main`: require PRs, require passing status checks, require linear history, block force-pushes. No one — not even you — pushes direct to `main`.
- Writes `.github/workflows/devx-ci.yml` (runs on PRs to `develop` and on pushes to `develop`).
- Writes `.github/workflows/devx-deploy.yml` (runs on pushes to `main` only).
- Writes `.github/workflows/devx-promotion.yml` (runs the extended gate; triggered by a `promote` label or by PromotionAgent).

## Parallelism & worktrees

At any given moment there can be multiple agents running. Each one owns a **worktree** on its own feature branch:

```
<repo>/                          ← main worktree, on `develop`, where the user is
  DEV.md, PLAN.md, TEST.md, ... ← canonical backlog state (on develop)
  dev/, plan/, test/, ...        ← canonical spec files (on develop)
.worktrees/
  dev-a3f2b9/                    ← DevAgent-7, branch develop/dev-a3f2b9
  dev-e2c7d1/                    ← DevAgent-3, branch develop/dev-e2c7d1
  debug-c4a7e2/                  ← DebugAgent-2, branch develop/debug-c4a7e2
```

### Rules

- Agents **never** share a worktree.
- Every feature branch is parented on `develop` (`git worktree add .worktrees/<name> -b develop/<type>-<hash> develop`).
- Backlog files live on `develop` in the main worktree. Agents in a worktree write status to their own spec file (in the main worktree, via a path-relative edit), and file new items by opening a trivial PR against `develop` that only touches the backlog file + spec file.
- Merges of agent work happen through PRs into `develop`. PRs must go green in CI before auto-merge.
- **No agent ever merges to `main`.** That's the promotion gate's job, period.
- ManageAgent decides capacity: the default cap is 3 simultaneous agents (configurable in `devx.config.yaml`).

### Why worktrees vs. checkpoints

Worktrees keep each agent's filesystem view stable and isolated. Two DevAgents working different items can't step on each other's working tree; there's no file-lock contention, no mid-edit rebase pain, and a crashed agent doesn't poison the main working directory. Branches + PRs handle the merge story the way git already handles it for humans.

## The `/devx-init` experience

This is where most users will form their first impression of devx. Raw BMAD's full menu is overwhelming for a first-time user. `/devx-init` must be the **simple guy to talk to** — a 5-question interview that gets you on the rails without forcing you to understand PRD vs. architecture vs. sprint-status on day one.

### The 8 questions

1. **What are you building?** — one sentence. Drives the initial `PLAN.md` seed and the first PRD draft.
2. **Who are you building it for?** — free-form archetypes, or "propose some." Drives focus-group panel creation ([`FOCUS_GROUP.md`](./FOCUS_GROUP.md)). `/devx-init` expands answers into 4–6 full persona profiles under `focus-group/personas/` before finishing, so the panel is ready the moment `/devx-plan` starts.
3. **Is any user's data in this project something you'd feel bad losing?** — "no" / "a few beta users" / "yes, real users." Sets risk **mode** ([`MODES.md`](./MODES.md)): YOLO / BETA / PROD. Detected existing repos with production signals default to PROD and ask for confirmation.
4. **What shape is this project in?** — empty-dream / bootstrapped-rewriting / mature-refactor-and-add / mature-yolo-rewrites / production-careful. Sets `project.shape` ([§ Project shapes](#project-shapes-orthogonal-to-mode)). Detected from repo state when not asked.
5. **Solo or team?** — solo, small team, shipping to end users, internal tool. Shapes which BMAD personas get prioritized and whether onboarding primitives for a second dev are scaffolded.
6. **Stack?** — optional but useful. Detected automatically from an existing repo (package.json, Cargo.toml, pubspec.yaml, …). If empty repo, ask.
7. **Infra preferences?** — GitHub Actions vs. other CI, Playwright vs. other browser harness, logs/metrics provider (or "wire later"). Defaults are opinionated: GitHub Actions, Playwright, "wire later" for observability.
8. **What's the first slice?** — one sentence about the first shippable thing. This becomes the first spec file in `DEV.md`, so the user can immediately run `/devx` and see the loop work end-to-end.

### What it sets up

- BMAD install (`core` + `bmm` + `tea`).
- Eight backlog files + the six spec subdirectories.
- `focus-group/` directory with `personas/`, `sessions/`, `prompts/`, and `index.md` (see [`FOCUS_GROUP.md`](./FOCUS_GROUP.md)).
- `.github/workflows/devx.yml` with lint + test + coverage gates.
- `.github/pull_request_template.md` with a spec-file link.
- `devx.config.yaml` — observability endpoints, capacity cap, branch prefix, worktree root, coverage policy.
- `.gitignore` updates for `.worktrees/`, `.devx-cache/`.
- `git` hook: `pre-push` coverage check.
- Initial `CLAUDE.md` if not present, with project context the agents will need.
- First spec file in `dev/` seeded from question #5, linked in `DEV.md`.

### What it does NOT do

- Doesn't run `/devx-plan`. That's the user's next call (or they can let `/devx-manage` schedule it).
- Doesn't push anything. Everything stays local until the user approves the first push.
- Doesn't invent product decisions. If something is ambiguous, it goes into `INTERVIEW.md`, not into a silent default.

### Existing projects vs. empty repos

Same command, different intake:

- **Empty repo:** all 5 questions asked, all scaffolding written.
- **Existing repo:** skip stack-detection questions, ask about the backlog ("we see recent commits on feature-X — is that in-flight? Should it get its own spec file?"), detect existing CI ("we see `.github/workflows/test.yml` — shall we extend it or replace it?"), inventory existing tests for coverage baseline.
- **Re-run (project already on devx):** upgrade paths — detect current devx version, apply migrations for new backlog fields, re-scaffold missing pieces. No-op if already current.

## Trust-gradient autonomy ladder

Progressive autonomy tied to the project's risk **mode** (see [`MODES.md`](./MODES.md)). A static "every promotion needs approval" gate becomes nagware; full autopilot erodes trust; mode-derived thresholds handle both.

**Default schedule per mode** (configurable in `devx.config.yaml` → `promotion.autonomy.n`):

| Mode | Initial N | After rollback | After hotfix-revert on main |
|---|---|---|---|
| YOLO | 0 (full autonomy immediately) | stays 0 | n/a |
| BETA | 3 | halves | zeros |
| PROD | 10 | halves | zeros |
| LOCKDOWN | ∞ (frozen) | — | — |

Until N promotions have accrued, every `develop → main` promotion requires user approval. After N, auto-promotion on all-gates-green; user retains veto window (revert commits within 24h).

User can freeze autonomy at any level with `devx autonomy --freeze <level>` or fully disable with `devx autonomy --off`. Mode changes override both — switching to LOCKDOWN instantly freezes, returning to the prior mode restores.

## Agent coordination primitives (preventing the "rip-through" race)

A real bug observed during dogfood: one `/dev` agent waits on CI, a *parallel* `/dev` agent picks up another `ready` item from `DEV.md`, and they collide — the second agent's scheduled wakeup fires while the first hasn't checked in yet, queues land out of order, branch state diverges.

The fix is explicit coordination primitives in `.devx-cache/`:

### Coordination files (gitignored)

```
.devx-cache/
├── locks/
│   ├── manager.lock             ← held by the active ManageAgent (single-writer)
│   ├── concierge.lock           ← held by the active ConciergeAgent (single-writer)
│   ├── spec-<hash>.lock         ← held by the agent currently working that spec
│   └── ci-wait-<branch>.lock    ← held by the agent waiting on CI for that branch
├── heartbeat.json               ← ManageAgent's liveness pulse
├── events/<agent-id>.jsonl      ← per-agent event stream
└── intents/<agent-id>.json      ← what an agent is about to do (read by Manager before it spawns parallel work)
```

### Rules

1. **ManageAgent holds `manager.lock` while it's making spawn decisions.** No other Manager instance starts; if found stale (heartbeat older than 5 min), force-takeover writes a `learn/` entry.
2. **ConciergeAgent holds `concierge.lock`** for the same reason — only one user-facing front door per machine.
3. **An agent acquires `spec-<hash>.lock` before claiming a `DEV.md` item.** A second agent finding the lock skips the item. Released on completion or timeout (default 2 hours).
4. **`ci-wait-<branch>.lock` held by the agent waiting on remote CI.** Manager sees this lock and **does not** spawn another agent that might race the same branch (e.g., a CI-fail-then-retry path). The waiting agent is the only owner of the branch until it releases.
5. **`intents/<agent-id>.json` declares the next operation** before it starts. Manager reads intents during scheduling; conflicts surface as deferrals.
6. **`ScheduleWakeup` and `Monitor`-style polling** are coordinated via the lock files, not via wall clock. An agent that wakes up to find its lock has been forcibly released stops; it doesn't blindly continue.

### Specific repro for the rip-through bug

- Agent A runs `/dev dev-X`, pushes branch, calls `ScheduleWakeup` for 120s to check CI.
- Manager tick fires; sees `ci-wait-<branch>.lock` is missing (Agent A didn't take it), thinks the branch is idle, spawns Agent B on `dev-Y` which depends transitively.
- Agent B's CI run interleaves with Agent A's; merge ordering becomes nondeterministic.

The fix: every agent waiting on CI **must** hold `ci-wait-<branch>.lock` for the duration of its wait, and Manager **must** read intents + locks before spawning anything that could touch the same branch family. This is mandatory wiring in `/devx` Phase 7 and in `/devx-manage`'s scheduling loop.

### Why filesystem locks (and not, say, a real DB)

Files give us: atomic create-if-missing via `O_EXCL`, naturally git-ignored, inspectable with `ls`, killable with `rm`, debuggable with `cat`. Same reasoning that put backlogs in markdown files. No new infra.

## Project shapes (orthogonal to mode)

Mode covers risk-to-user-data. **Project shape** covers state-of-codebase. They're independent and stack:

| Shape | Description | Default mode bias | Behavior changes |
|---|---|---|---|
| `empty-dream` | Empty directory, just an idea | YOLO | PlanAgent is permissive about scope expansion; tests written alongside, not before; aggressive parallelism |
| `bootstrapped-rewriting` | Has commits, but you'll be rewriting big chunks | YOLO/BETA | DevAgent prefers wholesale rewrites over surgical edits; no backward-compat constraints; coverage targets relaxed |
| `mature-refactor-and-add` | Shipped, stable, only small features and refactors | BETA | DevAgent must keep changes surgical; existing tests are sacrosanct (none deleted without explicit approval); blast radius capped per spec |
| `mature-yolo-rewrites` | Shipped but the user wants/expects big refactors | YOLO | Hybrid — coverage stays strict but DevAgent is allowed to delete/rewrite freely; promotion gate hardened to compensate |
| `production-careful` | Real users, every change matters | PROD | Maximum gates; DevAgent must produce migration plans + rollback plans for every spec; focus-group binding |

### How shape gets set

`/devx-init` adds a question (Q3.5, between mode and stack):

> **devx:** What shape is this project in?
> 1. Empty — just an idea.
> 2. Bootstrapped — has code, but you expect to rewrite big chunks.
> 3. Mature — shipping, only small refactors and additions wanted.
> 4. Mature but bold — shipping, but you're fine with big rewrites.
> 5. Production — real users; every change must be careful.

Detected from repo state when re-running init: empty dir → 1; commits but no tests → 2; commits + tests + tags → 3 or 5 depending on mode answer.

Stored in `devx.config.yaml → project.shape`.

### Why shape is its own axis (not a mode flavor)

A `mature-yolo-rewrites` project can still be in YOLO mode (no real users) and still want strict coverage and rewrite freedom — that's three independent levers. Collapsing them into one made `/devx-init` pick the wrong defaults for at least one project Leonid would actually run. Two axes is the minimum that disambiguates.

DevAgent reads both `project.shape` and `mode` and combines them: e.g., `mature-refactor-and-add + PROD` → "no test deletions, surgical diffs, migration plan required, focus-group binding pre-merge."

## Staying alive

ManageAgent and ConciergeAgent are long-running. The user wants them to keep going while away from the computer. **Three layers of resilience**, each independent so any one can fail and the others still hold:

### Layer 1: OS-level supervision

The `manage` and `concierge` processes are supervised by the OS init system, not by the user's terminal. Survives terminal close, machine sleep, even SSH disconnect.

- **macOS**: `~/Library/LaunchAgents/dev.devx.manager.plist` — LaunchAgent, auto-restarts on crash, starts at login.
- **Linux**: `~/.config/systemd/user/devx-manager.service` — systemd user unit, `Restart=on-failure`, `WantedBy=default.target`.
- **Windows / WSL**: Task Scheduler with restart-on-fail.

`/devx-init` writes the right one based on `uname`. Both processes log to `~/Library/Logs/devx/` (or `$XDG_STATE_HOME/devx/`).

### Layer 2: Mutual watchdog

ManageAgent and ConciergeAgent watch each other:

- ManageAgent writes `.devx-cache/heartbeat.json` every 60s.
- ConciergeAgent reads it; if stale > 5min, it spawns a fresh ManageAgent and notifies the user.
- ManageAgent monitors Concierge's PID; if dead, respawns.
- A tiny `devx-revive` cron runs every 15 min via launchd/systemd, checks both heartbeats, force-restarts whichever is stale. Breaks the cycle if both wedge.

### Layer 3: Cloud watchdog (away-from-laptop)

For hours-long absences, OS supervision isn't enough — laptop sleeps, network drops, process dies and nothing local notices. A GitHub Actions cron (`devx-cloud-watchdog.yml`) runs every 30 min:

- Polls a tiny status endpoint the local Concierge exposes (or, if unreachable, tails the last commit on `develop` for activity).
- If no agent activity in N hours and `DEV.md` has `ready` work, posts a `MANUAL.md` entry and FCM notification: *"devx looks idle on your machine. 3 ready items waiting. Wake the laptop?"*
- Optionally, if `cloud_spillover.enabled = true`, kicks the work to a Cloudflare Container or GitHub Actions runner instead of waking the user.

### Crash recovery

When ManageAgent restarts, it does **not** retain in-memory state — it reconstructs from disk:

1. Read `.devx-cache/state/schedule.json` (Manager's last desired roster).
2. Read `.devx-cache/state/manager.json` (last running roster snapshot).
3. For each agent in the snapshot: check if its PID is alive. If yes, re-attach to its event stream. If no, check the spec's status log — last entry < 5min ago → respawn; else mark `status: orphaned` and re-decide on the next scheduling tick.
4. Resume normal scheduling tick.

Spec status log + schedule snapshot = full recovery state. No in-process memory is load-bearing.

### What happens when context rots mid-task

The exact failure mode the user already hits today:

1. Worker emits "stopping due to context rot" (or token usage > threshold) into its jsonl.
2. ManageAgent reads the line, kills the subprocess.
3. ManageAgent appends to the spec's status log: `<ts> — context rot at 87%, restarting`.
4. ManageAgent spawns `claude /devx-<role> <slug>` afresh.
5. The new worker reads the spec's frontmatter + status log + acceptance criteria + any in-progress branch state, picks up where the last one left off.
6. ConciergeAgent emits a soft notification (suppressible per `notifications.events`).

No human in the loop. No continuation snippet to paste.

### What ManageAgent emits to the mobile relay

ManageAgent transforms raw worker jsonl into mobile-appropriate events. Token-by-token output stays local; the relay only sees the high-level transitions:

```json
{"ts": "2026-04-23T15:42:11Z", "agent": "dev-7", "spec": "dev-a3f2b9",
 "event": "context_rot_detected", "ctx_pct": 87, "action": "restarting"}
```

Keeps mobile push volume sane and makes notifications feel like updates, not noise.

## Operator primitives

The trust-gradient is only credible if the user has primitives to inspect and intervene. v0.1 ships:

- `devx ui` — open the TUI dashboard.
- `devx serve` — start the local web dashboard.
- `devx ask <question>` — talk to ConciergeAgent directly.
- `devx tail <agent-id>` — streams `.devx-cache/events/<agent-id>.jsonl` to stdout. Color-coded by phase.
- `devx kill <agent-id>` — terminates the agent's subprocess and marks its spec file `status: killed`. Worktree stays for inspection; `--clean` removes it too.
- `devx restart <agent-id>` — kill + respawn against the same spec.
- `devx status` — summary: active agents, pending backlogs per file, usage headroom, heartbeat age.
- `devx pause` / `devx resume` — pause Manager (lets in-flight workers finish, blocks new spawns).
- `devx config <key> [value]` — get/set config keys (writes to `devx.config.yaml`). Full key reference: [`CONFIG.md`](./CONFIG.md).
- `devx eject` — uninstalls devx-specific state (`.devx-cache/`, `.worktrees/`, devx slash commands) while leaving backlogs, spec files, and BMAD artifacts intact. Lock-in escape hatch.

## Line-level touched-surface coverage

Coverage gate runs only on lines changed in the PR vs. `develop`.

```
touched_lines = `git diff --unified=0 origin/develop...HEAD`
covered_touched = intersect(touched_lines, coverage_report.covered_lines)
gate = (len(covered_touched) / len(touched_lines)) >= 1.00
```

Opt-out per line: `# devx:no-coverage <reason>` (or language-appropriate comment prefix).

Auto-detected flaky tests: any test that fails then passes within 24h on the same commit triggers a `TEST.md` entry invoking the TEA `testarch-test-review` workflow.

## Fast-ship vs. careful promotion gates

The promotion-gate behavior is derived from the project **mode** (see [`MODES.md`](./MODES.md)) but can be explicitly overridden:

| Mode | Default gate behavior |
|---|---|
| YOLO | `fast-ship-always` — CI green only |
| BETA | `fast-ship` — CI green + no reviewer-blocking comments |
| PROD | `careful` — fast-ship gates + 24h soak on `develop` + exploratory QA pass + no new `DEBUG.md` items filed against develop in last 12h + focus-group panel clear |
| LOCKDOWN | `manual-only` — no automation, user promotes with `/devx-promote --force` + decision record |

Override explicitly in `devx.config.yaml → promotion.gate` when needed. The trust-gradient ladder applies within the chosen gate profile.

## Over-tuning detector

LearnAgent tracks, rolling 7-day window:
- `user_edits_to_skills` (git commits touching `~/.claude/commands/devx-*.md` by user)
- `lesson_applications` (commits prefixed `devx-learn:`)

When `user_edits > lesson_applications`, adds a soft signal to `MANUAL.md`:
> "You've been hand-editing skills more than LearnAgent is applying lessons this week. Consider letting LearnAgent observe for a cycle, or explicitly recording your preferences via `/devx-learn --add`."

Prevents the failure mode where the user becomes the tuning layer instead of the product-building layer.

## Contract between devx and BMAD

devx treats BMAD as a library, not a cage. Specifically:

- devx commands **invoke BMAD workflows** (`workflow.xml`, `create-prd/workflow.yaml`, `dev-story/workflow.yaml`, etc.) the same way `/dev` and `/dev-plan` do today.
- devx **does not modify** files under `_bmad/`. BMAD upgrades are safe.
- devx **reads and writes** `_bmad-output/` in the same shape BMAD does. If you stopped using devx tomorrow, raw BMAD commands would still work against your artifacts.

This is the escape hatch: if devx is the wrong shape for a particular project, users can fall back to raw `/bmad-*` commands without losing any work.
