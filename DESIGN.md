# Design

This is the working design doc. It captures the shape of the system as we currently understand it; anything marked **[TBD]** is an open question and should also live in [`OPEN_QUESTIONS.md`](./OPEN_QUESTIONS.md).

---

## Core idea: close the loop with a filesystem graph

The existing `/dev` and `/dev-plan` commands are excellent in isolation — `/dev-plan` does research → PRD → party-mode refinement → emits epic slugs; `/dev` takes a slug and implements it with full discipline (CI, coverage, adversarial review). But **the user is the glue** between them. You run `/dev-plan`, you read the output, you decide which epic to run, you run `/dev <slug>`, you wait, you pick the next one.

devx replaces the human glue with a **filesystem graph**:

- Every unit of work is a file (`dev/dev-<hash>-<timestamp>-<slug>.md`).
- Every backlog is a file (`DEV.md`, `TEST.md`, …) that references spec files.
- Every agent reads the backlog for its role, picks the top runnable item, and appends its status to the spec file as it works.
- A supervisor agent (`/devx-triage`) rebalances the backlog, resolves conflicts, and decides what runs next when capacity opens.

Nothing persists outside the repo. `git log` is the audit trail. `grep` is the query engine.

## The eight backlog files

All live at the repo root. Each is a markdown file whose body is a bulleted list of spec-file references, plus some human-readable context at the top.

### `DEV.md` — features to build

Written by: PlanAgents (new items), DevAgents (progress updates), Triage (reprioritization).
Read by: DevAgents (pick next), TestAgents (know what just shipped).

Each entry:
```markdown
- [ ] `dev/dev-a3f2b9-2026-04-23T14:22-add-oauth-google.md`
  - Title: Add Google OAuth to login
  - From: plan/plan-9c1d4a-epic-auth.md
  - Status: ready
  - Blocked by: —
  - Owner: —   (picked up → DevAgent-7 when claimed)
```

Statuses: `ready`, `in-progress`, `blocked`, `in-review`, `done`, `deleted`.

### `PLAN.md` — planning work in flight

Written by: PlanAgents, Triage.
Read by: PlanAgents (avoid re-planning the same thing), user.

Tracks active planning: which epics are being drafted, what research is outstanding, which party-mode workshops are scheduled.

### `TEST.md` — test work

Written by: DevAgents (declares coverage gaps on commit), TestAgents (progress), FocusAgent (user flow gaps).
Read by: TestAgents.

```markdown
- [ ] `test/test-f8e2a1-oauth-callback-coverage.md`
  - Target: services/auth/oauth.py — google_callback handler
  - Gap: no test for expired-state rejection
  - From: dev-a3f2b9 (auto-generated by DevAgent on merge)
```

### `DEBUG.md` — things that are broken

Written by: anyone (DevAgent on CI red, FocusAgent on user friction, user manually, TestAgent on flake).
Read by: DebugAgents.

```markdown
- [ ] `debug/debug-c4a7e2-2026-04-23T09:14-login-500-on-safari.md`
  - Source: production logs — 47 occurrences in last 24h
  - Symptom: 500 on POST /auth/login from Safari 17
  - First seen: 2026-04-22
  - Repro: [TBD by DebugAgent]
```

### `FOCUS.md` — user signals

Written by: FocusAgent (synthesis of telemetry + interviews), user (direct feedback).
Read by: PlanAgents (what to plan next), DebugAgents (known user pain).

Not individual items — instead, rolling summaries:
- Top 5 user friction points this week
- Features requested ≥3 times
- Flows with >20% abandonment
- Paths where session time is anomalously high

Each summary points at the raw telemetry query that produced it (so it can be re-run).

### `INTERVIEW.md` — questions for the user

Written by: any agent (when blocked on a decision a human must make).
Read by: the user.

```markdown
- [ ] Q#4 (from DevAgent-7 on dev-a3f2b9)
  - Context: implementing Google OAuth callback
  - Question: Should the redirect URI use our root domain (example.com/auth/callback) or a dedicated subdomain (auth.example.com/callback)?
  - Blocks: dev-a3f2b9, plan-9c1d4a (epic-auth)
  - Options: (a) root domain, (b) subdomain, (c) make configurable
  - Agent recommendation: (c)
```

The user answers inline by checking the box and adding a `→ Answer: ...` line. `/devx-triage` detects answered questions and unblocks the waiting items.

### `MANUAL.md` — things only the user can do

Written by: any agent (when action requires a human — approve a cloud resource, add a secret, sign in to a third-party service, review a sensitive PR).
Read by: the user.

```markdown
- [ ] M#2 (from DevAgent-7 on dev-a3f2b9)
  - Action: Add GOOGLE_OAUTH_CLIENT_SECRET to GitHub Actions secrets
  - Why: CI will fail integration tests without it
  - Blocks: dev-a3f2b9 merging
  - How: https://github.com/<org>/<repo>/settings/secrets/actions
```

Distinction from `INTERVIEW.md`: interview = decision, manual = action. Agents can recommend either, but users execute `MANUAL.md` items.

### `LESSONS.md` — self-healing proposals

Written by: LearnAgent (detected patterns), user (hand-crafted lessons via `/devx-learn --add`).
Read by: TriageAgent (when deciding capacity), user (approve/reject pending lessons), mobile app (inbox).

Each entry points at a `learn/learn-<hash>-<ts>-<slug>.md` spec file that carries the full evidence chain: which triggers produced the lesson, what target it proposes to edit, the proposed diff, the confidence level, and (once applied) the resulting commit.

See [`SELF_HEALING.md`](./SELF_HEALING.md) for the complete contract — trigger signals, confidence gates per write target, canary runs for skill/prompt changes.

## Spec file convention

Every item in every backlog points at a file. The path encodes type + identity + time + human-readable name:

```
<type>/<type>-<hash>-<timestamp>-<slug>.md

Examples:
  dev/dev-a3f2b9-2026-04-23T14:22-add-oauth-google.md
  test/test-f8e2a1-2026-04-23T15:50-oauth-callback-coverage.md
  debug/debug-c4a7e2-2026-04-23T09:14-login-500-on-safari.md
  plan/plan-9c1d4a-2026-04-20T10:00-epic-auth.md
  focus/focus-e1b5d8-2026-04-22T00:00-weekly-friction-summary.md
```

- `<type>` — one of `dev`, `plan`, `test`, `debug`, `focus`, `learn`. Implies the agent class.
- `<hash>` — 6 random hex chars. Unique handle; short enough to reference inline.
- `<timestamp>` — ISO 8601 local time at creation, minute precision. Lets you sort chronologically without a DB.
- `<slug>` — kebab-case human description, trimmed to ~50 chars.

### Spec file contents

```markdown
---
hash: a3f2b9
type: dev
created: 2026-04-23T14:22:00-07:00
title: Add Google OAuth to login
from: plan/plan-9c1d4a-epic-auth.md
spawned:
  - test/test-f8e2a1-oauth-callback-coverage.md
status: in-progress
owner: DevAgent-7
branch: devx/dev-a3f2b9
worktree: .worktrees/dev-a3f2b9
---

## Goal
<what this item does>

## Acceptance criteria
- [ ] user can click "Sign in with Google"
- [ ] callback validates state parameter
- [ ] valid callback creates/fetches user and returns JWT
- [ ] expired state returns 400

## Technical notes
<anything the agent figured out during implementation>

## Status log
- 2026-04-23T14:22 — created by PlanAgent-3
- 2026-04-23T14:25 — claimed by DevAgent-7
- 2026-04-23T14:28 — blocked on INTERVIEW q#4 (redirect URI)
- 2026-04-23T15:10 — unblocked (user answered: c, make configurable)
- 2026-04-23T15:45 — implementation complete, PR #142 opened
- 2026-04-23T15:50 — test gaps written to test/test-f8e2a1-...

## Links
- PR: https://github.com/.../pull/142
- CI: https://github.com/.../actions/runs/...
```

The status log is **append-only**. Agents add lines; they don't rewrite them. This is the request history — where a thing came from, where it went.

## Agent roles

Every agent class maps 1:1 to a slash command, but the slash commands can also be invoked manually. Triage spawns agents behind the scenes.

### `PlanAgent` — `/devx-plan`

- Input: a raw idea, or an entry in `PLAN.md`, or a signal from `FOCUS.md`.
- Work: research → PRD update → architecture update → epic chunking → party-mode refinement.
- Output: new spec files under `dev/`, new entries in `DEV.md`, questions in `INTERVIEW.md`, cross-refs from `FOCUS.md`.
- Inherits the discipline of the current `/dev-plan` command — the seven-phase loop, party-mode for every epic, three-layer coverage, YOLO vs. ask rules.

### `DevAgent` — `/devx`

- Input: next `ready` item in `DEV.md`.
- Work: create worktree + branch → implement in red-green-refactor → self-review (adversarial) → push → wait for CI → merge (or revise).
- Output: merged code, updated spec file status log, test gaps appended to `TEST.md`, debug items appended to `DEBUG.md` on CI failure.
- Inherits the discipline of the current `/dev` command — the six-phase loop, story-per-commit, check-models drift detection, pubspec bumps, CI-as-truth.

### `TestAgent` — `/devx-test`

- Input: next item in `TEST.md`.
- Work: author unit/integration/e2e tests, run browser agent for user flows, enforce 100% coverage on touched surface.
- Output: merged test code, regression items appended to `DEBUG.md` when a test reveals a bug.
- Leans on the BMAD `tea` module (test architecture) for strategy, ATDD, trace, NFR.

### `DebugAgent` — `/devx-debug`

- Input: next item in `DEBUG.md`.
- Work: pull logs/metrics/DB state via observability hooks → reproduce → fix → write regression test → push.
- Output: merged fix, test added to `TEST.md` progress, status log updated on the debug spec.

### `FocusAgent` — `/devx-focus`

- Input: raw user telemetry + `INTERVIEW.md` answered questions + direct user feedback.
- Work: synthesize patterns → cross-reference with planned work → identify gaps.
- Output: rolling summary in `FOCUS.md`, new items in `DEV.md` (feature requests), new items in `DEBUG.md` (friction spots), questions in `INTERVIEW.md` when synthesis is ambiguous.

### `LearnAgent` — `/devx-learn`

- Input: request graph (every spec file's status log + `git log` + recorded CI outcomes + skill-edit log).
- Work: scan for trigger signals (repeated blockers, repeated CI failures, user edits after agent writes, recurring INTERVIEW answers, repeated PR-review findings), cluster into lessons, draft diffs against the chosen write target, gate by confidence + blast radius.
- Output: `LESSONS.md` entries + `learn/*.md` spec files with full evidence. Auto-applies high-confidence + low-blast lessons; queues everything else for user approval. Canary-runs agent-prompt changes before merge.
- Scheduled by TriageAgent when capacity is idle and signals have accumulated; can also be invoked directly by the user or via `/loop 24h /devx-learn`.
- Operates under the same worktree + `develop/learn-<hash>` branching rules as every other agent.

Full contract: [`SELF_HEALING.md`](./SELF_HEALING.md).

### `TriageAgent` — `/devx-triage`

- Input: every backlog file.
- Work:
  1. Reconcile duplicates (two `DEV.md` entries describing the same feature → merge).
  2. Detect blocks (DevAgent-7 filed INTERVIEW q#4 → ensure dev-a3f2b9 shows `blocked`).
  3. Unblock on user answers (user checked q#4 → mark dev-a3f2b9 `ready` again).
  4. Reprioritize based on FOCUS signals.
  5. Schedule LearnAgent when trigger signals have accumulated and no active agent is running.
  6. Decide what to run when capacity opens: spawn the right agent class against the right spec file in the right worktree.
- Output: reordered backlog files, agent spawn decisions.
- Never touches code. Never runs tests. Only coordinates.

## Branching model

devx is opinionated on a **`develop/main` split**:

- **`main`** = production. What's deployed. Protected (branch protection enforced, no direct pushes, no force-pushes ever). Only accepts PRs from `develop`, and only via explicit promotion.
- **`develop`** = integration. Where in-progress agent work accumulates. This is where the mobile app adds items, where agent PRs merge, where coverage gates run, where e2e tests live.
- **Feature branches** = `develop/<type>-<hash>`, one per spec file. Branched off `develop`, PR'd back into `develop`.

### Branch naming

```
main                                           ← production, deployed
develop                                        ← integration, WIP
develop/dev-a3f2b9                             ← DevAgent-7 working dev-a3f2b9
develop/debug-c4a7e2                           ← DebugAgent-2 working debug-c4a7e2
develop/test-f8e2a1                            ← TestAgent-4 working test-f8e2a1
develop/plan-9c1d4a                            ← PlanAgent-3 working plan-9c1d4a
```

### Lifecycle of a unit of work

```
┌──────────────────┐    ┌───────────┐    ┌───────────────────┐    ┌────────┐
│ feature branch   │ →  │  develop  │ →  │  promotion gate   │ →  │  main  │
│ develop/dev-XXX  │    │ (CI green)│    │  (extended gates) │    │(deploy)│
└──────────────────┘    └───────────┘    └───────────────────┘    └────────┘
      agent works           auto-merge        user-approved
      in worktree           after CI          or Triage-decided
                            + review
```

Five stages:

1. **Agent works in its feature branch's worktree.**
2. **Agent opens PR → `develop`.** CI runs; code review runs; coverage gate runs.
3. **Auto-merge to `develop`** once gates pass. Triage can merge; no human required for this step by default.
4. **Promotion gate.** Separate, slower cadence. Runs extended checks: integration tests against staging, browser-agent QA re-walks critical user flows, 24h soak on develop, FOCUS telemetry shows no regressions. Can be user-gated or Triage-gated (configurable).
5. **Promote `develop → main`** via a squash-merge or merge commit. This push triggers the deploy pipeline.

### What each branch means for agents

- **DevAgents, DebugAgents, TestAgents, PlanAgents** → always branch off `develop`, always PR into `develop`. They literally cannot target `main`.
- **The mobile app** → commits directly to `develop` (backlog appends) or opens branches off `develop`. Never touches `main`.
- **The user hand-editing backlog files** → edits `develop` (that's the default branch agents and the phone read from). When the user wants to promote, they run `/devx-triage --promote` or merge the PR manually.
- **A dedicated PromotionAgent** (new, resolves part of OPEN_QUESTIONS #7) → runs the promotion gate logic. Opens the `develop → main` PR, runs extended checks, merges when they pass. User can veto by closing the PR.

### Why this shape

- **The phone can't brick production.** Mobile writes hit `develop`; if they're bad, agent CI catches it before `main`.
- **Agents iterate fast on `develop`.** No wait for production deploy on every DevAgent merge.
- **Deploy cadence is a separate decision from merge cadence.** You can merge 30 agent PRs to `develop` before deciding any of them are ready to ship.
- **Rollback is `git revert` on `main`, not on the whole work graph.** Production is isolated from WIP.
- **CI configuration is layered:** `develop` CI = lint + unit + integration + coverage. `main` CI = deploy + smoke. Different jobs, different budgets.

### What `/devx-init` sets up for this

- Creates `develop` branch if absent, sets it as the default branch for the repo.
- Enables branch protection on `main`: require PRs, require passing status checks, require linear history, block force-pushes. No one — not even you — pushes direct to `main`.
- Writes `.github/workflows/devx-ci.yml` (runs on PRs to `develop` and on pushes to `develop`).
- Writes `.github/workflows/devx-deploy.yml` (runs on pushes to `main` only).
- Writes `.github/workflows/devx-promotion.yml` (runs the extended gate; triggered by a `promote` label or by PromotionAgent).

## Parallelism & worktrees

At any given moment there can be multiple agents running. Each one owns a **worktree** on its own feature branch:

```
<repo>/                          ← main worktree, on `develop`, where the user is
  DEV.md, PLAN.md, TEST.md, ... ← canonical backlog state (on develop)
  dev/, plan/, test/, ...        ← canonical spec files (on develop)
.worktrees/
  dev-a3f2b9/                    ← DevAgent-7, branch develop/dev-a3f2b9
  dev-e2c7d1/                    ← DevAgent-3, branch develop/dev-e2c7d1
  debug-c4a7e2/                  ← DebugAgent-2, branch develop/debug-c4a7e2
```

### Rules

- Agents **never** share a worktree.
- Every feature branch is parented on `develop` (`git worktree add .worktrees/<name> -b develop/<type>-<hash> develop`).
- Backlog files live on `develop` in the main worktree. Agents in a worktree write status to their own spec file (in the main worktree, via a path-relative edit), and file new items by opening a trivial PR against `develop` that only touches the backlog file + spec file.
- Merges of agent work happen through PRs into `develop`. PRs must go green in CI before auto-merge.
- **No agent ever merges to `main`.** That's the promotion gate's job, period.
- Triage decides capacity: the default cap is 3 simultaneous agents (configurable in `devx.config.yaml`).

### Why worktrees vs. checkpoints

Worktrees keep each agent's filesystem view stable and isolated. Two DevAgents working different items can't step on each other's working tree; there's no file-lock contention, no mid-edit rebase pain, and a crashed agent doesn't poison the main working directory. Branches + PRs handle the merge story the way git already handles it for humans.

## Request history (traceability)

The chain from idea → shipped is recoverable from files alone:

```
focus/focus-e1b5d8-weekly-friction-summary.md
  ↓ FocusAgent identified "login frustration — 12% abandon"
  ↓ (logged: "feature request: simplify login")
plan/plan-9c1d4a-epic-auth.md
  ↓ PlanAgent read FOCUS, drafted epic-auth
  ↓ (spawned: dev-a3f2b9, dev-e2c7d1, dev-f1b9d0)
dev/dev-a3f2b9-add-oauth-google.md
  ↓ DevAgent-7 built it
  ↓ (merged: PR #142)
test/test-f8e2a1-oauth-callback-coverage.md
  ↓ TestAgent filled the gap DevAgent declared
  ↓ (merged: PR #143)
debug/debug-c4a7e2-login-500-on-safari.md
  ↓ DebugAgent-2 found and fixed a regression
  ↓ (merged: PR #151)
```

Each link in the chain is a frontmatter field (`from:`, `spawned:`) plus a `git log` entry. You can `grep -r "a3f2b9" .` and find every file in the graph that touched this request.

## The `/devx-init` experience

This is where most users will form their first impression of devx. Raw BMAD's full menu is overwhelming for a first-time user. `/devx-init` must be the **simple guy to talk to** — a 5-question interview that gets you on the rails without forcing you to understand PRD vs. architecture vs. sprint-status on day one.

### The 5 questions

1. **What are you building?** — one sentence. Drives the initial `PLAN.md` seed and the first PRD draft.
2. **Who is it for?** — solo, small team, shipping to end users, internal tool. Drives which BMAD personas get prioritized and whether `/devx-focus` is wired up day one.
3. **Stack?** — optional but useful. Detected automatically from an existing repo (package.json, Cargo.toml, pubspec.yaml, …). If empty repo, ask.
4. **Infra preferences?** — GitHub Actions vs. other CI, Playwright vs. other browser harness, logs/metrics provider (or "wire later"). Defaults are opinionated: GitHub Actions, Playwright, "wire later" for observability.
5. **What's the first slice?** — one sentence about the first shippable thing. This becomes the first spec file in `DEV.md`, so the user can immediately run `/devx` and see the loop work end-to-end.

### What it sets up

- BMAD install (`core` + `bmm` + `tea`).
- Seven backlog files + the five spec subdirectories.
- `.github/workflows/devx.yml` with lint + test + coverage gates.
- `.github/pull_request_template.md` with a spec-file link.
- `devx.config.yaml` — observability endpoints, capacity cap, branch prefix, worktree root, coverage policy.
- `.gitignore` updates for `.worktrees/`, `.devx-cache/`.
- `git` hook: `pre-push` coverage check.
- Initial `CLAUDE.md` if not present, with project context the agents will need.
- First spec file in `dev/` seeded from question #5, linked in `DEV.md`.

### What it does NOT do

- Doesn't run `/devx-plan`. That's the user's next call (or they can let `/devx-triage` do it).
- Doesn't push anything. Everything stays local until the user approves the first push.
- Doesn't invent product decisions. If something is ambiguous, it goes into `INTERVIEW.md`, not into a silent default.

### Existing projects vs. empty repos

Same command, different intake:

- **Empty repo:** all 5 questions asked, all scaffolding written.
- **Existing repo:** skip stack-detection questions, ask about the backlog ("we see recent commits on feature-X — is that in-flight? Should it get its own spec file?"), detect existing CI ("we see `.github/workflows/test.yml` — shall we extend it or replace it?"), inventory existing tests for coverage baseline.
- **Re-run (project already on devx):** upgrade paths — detect current devx version, apply migrations for new backlog fields, re-scaffold missing pieces. No-op if already current.

## Relationship to BMAD output artifacts

`_bmad-output/planning-artifacts/` and `_bmad-output/implementation-artifacts/` are still the source of truth for **detailed** planning — the full PRD, the architecture diagrams, the sprint-status yaml. devx's backlog files are a **lightweight index** on top of them.

- A `PLAN.md` entry points at `plan/plan-9c1d4a-epic-auth.md`, which in turn references `_bmad-output/planning-artifacts/epic-auth.md` (the big BMAD-shaped epic file).
- A `DEV.md` entry points at `dev/dev-a3f2b9-*.md`, which in turn references `_bmad-output/implementation-artifacts/story-a3f2b9.md` (the big BMAD-shaped story file).

This split exists because BMAD's artifacts are information-dense (good for agents doing the work) but hard to scan at a glance (bad for triage). The devx spec files are scan-first; the BMAD artifacts are the full details.

## Contract between devx and BMAD

devx treats BMAD as a library, not a cage. Specifically:

- devx commands **invoke BMAD workflows** (`workflow.xml`, `create-prd/workflow.yaml`, `dev-story/workflow.yaml`, etc.) the same way `/dev` and `/dev-plan` do today.
- devx **does not modify** files under `_bmad/`. BMAD upgrades are safe.
- devx **reads and writes** `_bmad-output/` in the same shape BMAD does. If you stopped using devx tomorrow, raw BMAD commands would still work against your artifacts.

This is the escape hatch: if devx is the wrong shape for a particular project, users can fall back to raw `/bmad-*` commands without losing any work.
