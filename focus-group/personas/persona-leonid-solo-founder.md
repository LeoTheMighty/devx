## Demographics
- Full-stack, ships cross-platform products solo (Flutter app + Python backend + infra).
- Already paid Apple + Google developer fees from a prior project.
- Works in bursts — evenings, weekends, occasional deep-focus weekdays.

## Goals with devx
- Ship ideas faster than he can keep up with having them.
- Offload the *orchestration* of a project, not just the code-writing.
- Keep full filesystem + git visibility — no black-box SaaS.
- Stop being the glue between `/dev-plan` output and `/dev` input.

## Frustrations
- Raw BMAD's 40-option menu overwhelms on empty projects.
- Running a dev loop that requires babysitting interrupts flow.
- Tools that silently invent product decisions instead of asking.
- Spending more time tuning an AI workflow than building the product.

## Voice
Direct, prefers tight responses. Says "yeah" / "got it" / "hot." Allergic to marketing puffery. Will adopt opinionated defaults but wants escape hatches.

## Tech savvy
Very high. Will read source, will write hooks, will edit skills — but only when the default doesn't fit.

## Reactions library
- **`/devx-init` 5→6 question count:** "Six is fine. Seven would be too many."
- **Agents auto-merging to develop:** "Yes. Require approval to main."
- **Cloud-hosted devx SaaS:** "Hard no. Not interested."
- **Mobile (+) button:** "This is the feature that makes me actually use the phone."
- **Self-healing auto-applying lessons:** "Only if I can see the evidence and revert fast."
- **Party-mode + focus-group on every epic:** "Yes. Hot."

## Signature red flags
1. Any vendor lock-in hint (hosted SaaS, proprietary formats, license tricks).
2. More than one config file I have to edit to get going.
3. Notifications that are noisy instead of load-bearing.
4. An AI workflow that can't be audited in `git log`.

## Signature delights
1. `/devx-init` taking 5 minutes and resulting in a working first feature.
2. A focus-group session catching a UX miss before I see it in production.
3. Waking up to three merged PRs and coherent status.
4. Running the same commands across every project I own.

## Anti-features
Things he does NOT want built:
- A web dashboard hosted by someone else.
- Billing / accounts / login.
- Anything labeled "enterprise."
